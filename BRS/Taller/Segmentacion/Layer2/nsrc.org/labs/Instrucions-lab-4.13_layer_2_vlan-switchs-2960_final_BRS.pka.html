Introdución

O propósito deste exercicio é crear VLAN separadas para diferentes grupos de usuarios en cada edificio. Nun exercicio posterior configuraremos o enrutador principal para que cada VLAN use unha subrede IP diferente.

Este laboratorio é unha continuación do Laboratorio de Spanning Tree Protocol e a configuración do laboratorio é idéntica (var apartado Topoloxía).

   

VLANs

Agora queremos segmentar a rede para separar o tráfico de xestión da rede do tráfico de usuarios finais (persoal e estudantes).

Executar unha gran rede plana en todo o campus simplemente non se escala como se cubriu durante a presentación. Cada un destes segmentos será unha subrede separada.

Temos que adoptar un enfoque estruturado con esta migración. Aínda que estamos a traballar nun laboratorio para este obradoiro, unha migración da rede de campus dunha rede plana a outra redireccionada necesita atención e planificación.

O proceso será o seguinte:

  1.  Crear unha VLAN para cada persoal (chamada STAFF)
  2.  Crear unha VLAN para estudantes (chamada ESTUDANTE)
  3.  Crear unha VLAN para a xestión de dispositivos (chamada MGMT)
  4.  Apagado de VLAN1: a VLAN1 é a predeterminada de Cisco, ten moitos riscos de seguridade e nunca se debe usar.
    NOTA: A VLAN 1 non debe usarse para a administración e debe crearse outra VLAN por motivos de seguridade. A VLAN 1 é a VLAN predeterminada na maioría dos switches e está asociada con outros servizos predeterminados. Isto significa que calquera dispositivo conectado a calquera porto de switch pode comunicarse con outros dispositivos noutros portos de switch.

    Usar VLAN 1 para a xestión faino vulnerable aos ataques. Un atacante que obteña acceso á VLAN 1 podería acceder á configuración do switch e, potencialmente, a toda a rede.

    A creación dunha VLAN separada para a xestión permite illar o tráfico de xestión do resto da rede, o que dificulta o acceso dos atacantes aos dispositivos de xestión. Tamén lle permite aplicar políticas de seguridade específicas á VLAN de xestión, como o control de acceso baseado en roles, que limita o acceso aos usuarios autorizados.

    Polo tanto, é unha boa práctica de seguridade crear unha VLAN separada para a xestión e non usar a VLAN 1 para este fin.

Deshabilitando VTP

VTP (VLAN Trunking Protocol) é unha tecnoloxía propietaria de Cisco que permite o aprovisionamento dinámico de VLAN. Non o usaremos aquí.

Execute os seguintes comandos na CLI de todos os switchs: 

> enable
# configure terminal
(config)# vtp mode transparent

Configure os portos troncais

O switch de núcleo conéctase aos switchs de distribución de cada edificio, os cales á súa vez conéctanse a dous switchs de borde en cada edificio. Para permitir o paso de etiquetas VLAN entre switchs, é necesario configurar as interfaces que os conectan como portos troncais. (A configuración predeterminada de Cisco é que as interfaces pasen tramas Ethernet sen etiquetar).

Facendo referencia ao diagrama de rede anterior, configure o seguinte para cada porto que necesite etiquetar tramas VLAN. Por exemplo, no switch de distribución en Building 1 no CampusX dist1-b1-campusX:

> enable
# configure terminal
(config)# interface fastEthernet 0/13
(config-if)# description Trunk Link to edge1-b1-campusX
(config-if)# switchport mode trunk
(config-if)# exit
(config)# interface fastEthernet 0/12
(config-if)# description Trunk Link to edge1-b1-campusX
(config-if)# switchport mode trunk
(config-if)# end
#

Nota: A configuración predeterminada de Cisco é usar a encapsulación dot1q (en lugar de ISL propietario de Cisco).

Recordatorio: Comprobe o diagrama para ver cales son os portos que precisa modificar.

E non esqueza poñer as descricións nas interfaces!

    Direccionamento IP para a VLAN MGMT

    Inicialmente configuramos os switchs de xeito que estabamos usando a VLAN 1 para xestionalos (e proporcionar acceso ao usuario final a toda a rede universitaria!). E o bloque de enderezos IPv4 que usamos ata agora era todo 172.20.0.0/16 xa que tiñamos un gran dominio de difusión para toda a rede universitaria.

    Temos que afastarnos disto agora.

    Non podemos asignar direccións ás VLANs que creamos utilizando unha subrede do rango 172.20.0.0/16, xa que todo o bloque de direccionamento está a ser utilizado pola VLAN 1. O IOS de Cisco non permite que esas subredes sexan configuradas nos dispositivos de enrutamento. (O switch core1-campusX será o encargado de enrutar o tráfico entre VLANs).

    Para este exercicio simplemente eliminaremos os enderezos IP de xestión da VLAN 1 e configuraremos novas subredes na VLAN 10 (MGMT para o Building 1) e na VLAN 20 (MGMT para o Building 2).

    (Nota: Se realizas isto nunha rede activa, eliminando o enderezo IP e desactivando a VLAN 1 eliminarase toda a conectividade a Internet de todos os usuarios conectados aos switchs, polo que esta actividade é mellor facela cando a rede non estea en uso, normalmente pola mañá ou na fin de semana, dependendo de cando a rede universitaria non estea moi activa.). Lembra que, aínda que podemos eliminar un enderezo IPv4 simplemente facendo "no ip address", temos que escribir todo o enderezo IPv6 para borralo, "no ipv6 address 2001:db8:X:0::1:1/64", porque IPv6 permite varios enderezos por interfaces, a diferenza de IPv4).

    Nota: É moi importante eliminar todos os enderezos IP (IPv4 e IPv6) da VLAN 1 e pechar a VLAN 1 antes de configurar calquera cousa nas novas VLAN de xestión (VLAN 10 ou VLAN 20).

    Que ves?

    O comando show interfaces proporciona unha visión detallada de todas as interfaces, incluíndo o estado (up/down), estatísticas de tráfico, erros e configuración básica; mentres que o comando show interfaces | include Description filtra a saída para mostrar só as liñas relacionadas coas descricións das interfaces, permitindo localizar rapidamente a información sobre as descricións configuradas sen ter que revisar toda a saída completa.

    É por iso que é importante asegurarse de que todas as interfaces teñan unha liña de descrición configurada: xa que os comandos de diagnóstico anterior permiten que calquera operador vexa dunha ollada que interfaces do switch se usan para que función.

Deberiamos executar no switch de nucleo para as interfaces Fa0/1 e Fa0/2: 'switchport trunk allowed vlan 10,11,12,20,21,22'

    3. Podes facer ping a algún dos switchs dos outros edificios da túa rede universitaria. É dicir, podes facer ping dende algún switch de Building1 a alǵun switch de Building2? NON

      Que está pasando? Pois 2 cousas: primeiro que non non somos quen de chegar ao switch de núcleo(explicado no apartado anterior) e segundo que o switch de núcleo non está a facer de gateway para poder redireccionar os paquetes ICMP do ping.

    Apéndice B - Switch L3 - Enrutamento

    Neste caso, dado que as VLAN dos switches de Building1 e Building2 pertencen a subredes diferentes (172.20.1X.0/24 e 172.20.2X.0/24, respectivamente), non habería conectividade directa entre elas sen unha intervención adicional. O switch de núcleo core1-campusX está configurado como un simple switch en modo Layer 2, e non pode enrutar tráfico entre VLANs ou subredes.

    Por que non hai conectividade actualmente?

    1. Os switches de Building1 e Building2 teñen redes diferentes, e o tráfico entre subredes distintas require un dispositivo de Capa 3 (como un router ou un switch configurado para facer enrutamento inter-VLAN).
    2. O switch core1-campusX, ao ser un 2960, necesita configuración adicional para realizar enrutamento inter-VLAN, xa que por defecto é un dispositivo de capa 2.

    Que habería que facer no switch core1-campusX?

    1. Habilitar o enrutamento inter-VLAN: Como o switch 2960 non está deseñado para ser un router, deberíamos sustituilo por outro modelo, como o Cisco Catalyst 3560 que pode configurarse para manexar tráfico entre VLANs, de tal forma que deberiamos neste switch activar o enrutamento inter-VLAN e asignarlle unha IP a cada interface VLAN.
    2. Configurar as interfaces VLAN no switch:
      • Asegúrate de que as interfaces VLAN no switch teñan direccións IP asignadas ás subredes respectivas e estean en estado up.
      • Configura as VLAN que correspondan ao tráfico entre os switches de Building1 e Building2.
    3. Verificar as configuracións de trunking: Asegúrate de que os portos Fa0/1 e Fa0/2 do switch core1-campusX están correctamente configurados como trunk para transportar todas as VLANs.
    4. Configurar rutas predeterminadas nos switches de Building1 e Building2:

      En cada switch, configura como gateway predeterminado a dirección IP da interface VLAN correspondente ao switch de núcleo.
